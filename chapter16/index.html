    Chapter 16. 우리 팀 웹앱 만들기 ① – 핵심 기능 구현¶

    🎯 이 장에서 배우는 것¶

    • 와이어프레임을 기반으로 AI에게 체계적인 구현 지시를 내릴 수 있다
    • "레고 블록 전략"으로 기능을 작은 단위로 나눠 구현할 수 있다
    • 웹앱의 핵심 기능 1~2개를 AI와 협업하여 실제로 만들 수 있다
    • 오류(버그)가 발생했을 때 AI에게 디버깅을 요청하는 방법을 알 수 있다

    💡 왜 이걸 배우나요?¶

    🤔 생각해보기: 14장에서 만든 와이어프레임과 13장의 순서도를 꺼내보세요. 이 설계도를 AI에게 어떻게 설명하면 좋을까요?

    여러분은 이미 무엇을 만들지(기획)와 어떻게 생겼는지(와이어프레임)를 정했습니다. 이제 드디어 실제로 동작하는 웹앱을 만들 차례입니다!

    하지만 "우리 팀 웹앱 전체를 만들어줘"라고 AI에게 한 번에 요청하면 어떻게 될까요? 거대하고 복잡한 코드가 쏟아져 나오고, 어디서 뭐가 잘못됐는지 찾을 수도 없게 됩니다.

    전문 개발자들도 프로그램을 한 번에 완성하지 않습니다. 작은 기능 하나씩, 마치 레고 블록을 조립하듯 쌓아갑니다. 이번 장에서는 이 "레고 블록 전략"을 배우고, AI와 함께 핵심 기능을 하나씩 구현해봅니다.

    📚 핵심 개념¶

    개념 1: 레고 블록 전략 – 한 번에 하나씩¶

    🧱 비유로 시작

    레고로 성(城)을 만든다고 생각해보세요. 설명서를 보면 1단계: 바닥판, 2단계: 벽, 3단계: 문, 4단계: 지붕… 이렇게 단계별로 조립합니다. 한꺼번에 모든 블록을 쏟아놓고 동시에 끼우려 하면 엉망이 되죠.

    📖 정확한 정의

    레고 블록 전략이란, 웹앱의 전체 기능을 독립적인 작은 단위(블록)로 나누고, 한 번에 하나의 블록만 AI에게 요청하여 구현하는 방법입니다.

    💻 예시로 확인

    "할 일 관리 웹앱"을 만든다면 이렇게 나눕니다.

    flowchart LR A["🧱 블록 1<br/>화면 레이아웃"] --> B["🧱 블록 2<br/>할 일 추가"] B --> C["🧱 블록 3<br/>할 일 목록 표시"] C --> D["🧱 블록 4<br/>완료 체크"] D --> E["🧱 블록 5<br/>삭제 기능"]

    📌 알아보기: 블록의 순서가 중요합니다! 화면이 없으면 버튼을 넣을 수 없고, 목록이 없으면 삭제할 것도 없습니다. 앞 블록이 뒤 블록의 토대가 됩니다.

    개념 2: 와이어프레임을 구현 지시문으로 바꾸기¶

    🗺️ 비유로 시작

    와이어프레임은 건축의 설계도입니다. 하지만 설계도를 시공팀(여기서는 AI)에게 건넬 때는 "여기에 창문을 넣어주세요, 크기는 1m × 1m, 재질은 이중유리"처럼 구체적인 지시서로 바꿔야 합니다.

    📖 정확한 정의

    구현 지시문이란, 와이어프레임의 각 요소를 AI가 코드로 변환할 수 있도록 위치, 동작, 데이터를 명확히 기술한 프롬프트입니다.

    💻 예시로 확인

    와이어프레임에 "입력창과 추가 버튼"이 있다면, 아래처럼 변환합니다.

    • ❌ 모호한 지시: "할 일을 추가하는 기능을 만들어줘"
    • ✅ 명확한 지시: "HTML, CSS, JavaScript로 만들어줘. 화면 상단에 텍스트 입력창(placeholder: '할 일을 입력하세요')과 '추가' 버튼을 나란히 배치해줘. 버튼을 클릭하면 입력한 텍스트가 아래 목록에 추가되고, 입력창은 비워져야 해. 빈 칸일 때 버튼을 누르면 '내용을 입력해주세요'라는 알림이 나와야 해."

    💡 정리하기: 좋은 구현 지시문의 3요소 - 기술 스택: 어떤 언어/도구를 쓸지 (HTML, CSS, JavaScript) - 화면 구성: 어디에 무엇이 보이는지 - 동작 설명: 사용자가 무엇을 하면 어떤 일이 일어나는지

    개념 3: 점진적 구현 – 확인하고 다음으로¶

    🏗️ 비유로 시작

    벽돌을 쌓을 때, 한 층을 올리면 수평이 맞는지 확인하고 다음 층을 올립니다. 확인 없이 10층까지 쌓았는데 3층이 기울어져 있다면? 전부 무너뜨리고 다시 해야 합니다.

    📖 정확한 정의

    점진적 구현이란, 하나의 블록을 구현할 때마다 실행하여 정상 동작을 확인한 후, 다음 블록으로 넘어가는 개발 방식입니다.

    💻 흐름 확인

    flowchart TD A["블록 1 구현 요청"] --> B["코드를 받아 실행"] B --> C{"정상 작동?"} C -->|"✅ Yes"| D["블록 2 구현 요청"] C -->|"❌ No"| E["오류 메시지를<br/>AI에게 전달"] E --> F["수정된 코드 받기"] F --> B D --> G["코드를 받아 실행"] G --> H{"정상 작동?"} H -->|"✅ Yes"| I["다음 블록으로..."] H -->|"❌ No"| J["오류 해결 반복"] J --> G

    📌 알아보기: 매번 확인하는 것이 느려 보이지만, 결국 가장 빠른 방법입니다. 오류를 일찍 발견할수록 고치기 쉽습니다!

    개념 4: AI에게 디버깅 요청하기¶

    🔍 비유로 시작

    병원에 갈 때 "아파요"라고만 하면 의사가 진단하기 어렵습니다. "어제부터 오른쪽 배 아래가 콕콕 쑤시고, 누르면 더 아파요"라고 말하면 훨씬 정확한 진단이 가능합니다. 오류 해결도 마찬가지입니다.

    📖 정확한 정의

    디버깅(debugging)이란 프로그램의 오류(버그)를 찾아 고치는 과정입니다. AI에게 디버깅을 요청할 때는 에러 메시지, 현재 코드, 기대한 동작을 함께 전달해야 합니다.

    💻 예시로 확인

    • ❌ 나쁜 디버깅 요청: "안 돼요, 고쳐주세요"
    • ✅ 좋은 디버깅 요청: "아래 코드에서 '추가' 버튼을 클릭하면 목록에 아무것도 추가되지 않아요. 콘솔에 TypeError: Cannot read properties of null (reading 'value') 에러가 나옵니다. 입력창의 텍스트를 읽어서 목록에 추가하는 것이 원래 의도입니다. [코드 전체를 붙여넣기]"

    💡 정리하기: 디버깅 요청 공식 - 증상: 무엇이 잘못되었는지 - 에러 메시지: 콘솔에 나온 빨간 글씨 그대로 복사 - 기대 동작: 원래 어떻게 되어야 하는지 - 현재 코드: 전체 코드 붙여넣기

    🔨 따라하기¶

    이번 실습에서는 "할 일 관리 웹앱"을 예시로 진행합니다. 여러분 팀의 프로젝트에 같은 전략을 적용하세요!

    Step 1: 기능을 블록으로 나누기¶

    14장 와이어프레임과 13장 순서도를 펼치고, 팀원과 함께 기능을 블록으로 나눕니다.

    🤔 생각해보기: 우리 팀 웹앱의 핵심 기능은 무엇인가요? 그중 가장 기본이 되는 1가지는?

    예시로 "할 일 관리 웹앱"의 블록을 나눠봅니다.

    • 블록 1: 기본 화면 구성 (제목, 입력창, 버튼, 목록 영역)
    • 블록 2: 할 일 추가 기능 (입력 → 버튼 클릭 → 목록에 표시)
    • 블록 3: 완료 체크 기능 (클릭하면 취소선 표시)
    • 블록 4: 삭제 기능 (X 버튼 → 목록에서 제거)

    오늘 목표: 블록 1과 블록 2를 완성합니다!

    Step 2: 블록 1 – 기본 화면 구성 요청¶

    AI에게 아래와 같이 요청합니다.

    HTML, CSS, JavaScript를 하나의 HTML 파일로 만들어줘.

[화면 구성]
- 화면 중앙에 최대 너비 500px인 카드 형태의 영역
- 상단에 "📋 할 일 관리" 제목 (h1 태그)
- 그 아래에 텍스트 입력창과 "추가" 버튼을 한 줄에 나란히 배치
  - 입력창 placeholder: "할 일을 입력하세요"
  - 버튼 배경색: 파란색, 글자색: 흰색
- 버튼 아래에 빈 목록 영역 (ul 태그, id="todo-list")

[스타일]
- 배경색: 연한 회색 (#f0f0f0)
- 카드에 그림자 효과
- 깔끔하고 둥근 느낌의 디자인

아직 JavaScript 기능은 넣지 말고, 화면 구조와 CSS만 만들어줘.

    Step 3: 코드를 받아 실행하고 확인하기¶

    AI가 생성한 코드를 복사하여 index.html 파일로 저장하고 브라우저에서 엽니다.

    확인 체크리스트: - ✅ 제목이 보이는가? - ✅ 입력창과 버튼이 나란히 있는가? - ✅ 카드 형태로 중앙에 정렬되었는가? - ✅ 디자인이 깔끔한가?

    문제가 있다면 구체적으로 AI에게 수정 요청합니다.

    입력창과 버튼이 나란히 배치되지 않고 위아래로 쌓여 있어.
입력창이 가로 공간의 70%를 차지하고, 버튼이 나머지 30%를 차지하도록
flexbox로 수정해줘.

    Step 4: 블록 2 – 할 일 추가 기능 요청¶

    화면이 정상이면, 이제 기능을 추가합니다. 기존 코드를 함께 보내는 것이 핵심입니다.

    아래는 현재까지 작성된 코드야. 여기에 JavaScript를 추가해서
"할 일 추가" 기능을 구현해줘.

[동작 설명]
1. 입력창에 텍스트를 적고 "추가" 버튼을 클릭하면
2. 아래 목록(ul#todo-list)에 새로운 항목(li)이 추가된다
3. 추가 후 입력창은 자동으로 비워진다
4. 입력창이 비어있을 때 버튼을 누르면 "할 일을 입력해주세요!" 알림(alert)을 띄운다
5. Enter 키를 눌러도 추가되도록 한다

(여기에 블록 1의 전체 코드를 붙여넣기)

    Step 5: 오류가 발생하면 – 디버깅 실습¶

    실행했는데 버튼을 눌러도 아무 반응이 없다면? 브라우저 개발자 도구를 열어봅니다.

    개발자 도구 여는 법: - Windows: F12 또는 Ctrl + Shift + I - Mac: Cmd + Option + I - "Console(콘솔)" 탭 클릭

    콘솔에 빨간색 에러 메시지가 보이면, 그대로 복사하여 AI에게 전달합니다.

    아래 코드에서 "추가" 버튼을 클릭하면 목록에 할 일이 추가되지 않아요.

[에러 메시지]
Uncaught TypeError: Cannot read properties of null (reading 'addEventListener')
at index.html:45

[기대하는 동작]
버튼을 클릭하면 입력한 텍스트가 목록에 추가되어야 합니다.

[전체 코드]
(전체 코드 붙여넣기)

    📌 알아보기: 위 에러는 JavaScript가 HTML 요소를 찾지 못할 때 자주 발생합니다. 보통 <script> 태그가 HTML 요소보다 먼저 실행되기 때문입니다. AI가 <script> 태그를 </body> 바로 위로 옮기거나, DOMContentLoaded 이벤트를 사용하도록 수정해줄 것입니다.

    Step 6: 완성 확인 및 팀 프로젝트 적용¶

    할 일 추가 기능이 정상 작동하면, 같은 전략을 우리 팀 프로젝트에 적용합니다.

    팀별 활동 순서:

    1. 우리 팀 와이어프레임에서 핵심 기능 블록 목록을 작성한다
    2. 가장 기본이 되는 블록 1개를 선택한다
    3. 구현 지시문을 작성하여 AI에게 요청한다
    4. 코드를 실행하고 확인한다
    5. 오류가 있으면 디버깅 요청 → 수정 → 재확인
    6. 정상 작동하면 다음 블록으로 넘어간다

    📝 전체 코드¶

    아래는 블록 1(화면 구성)과 블록 2(할 일 추가)가 완성된 전체 코드입니다.

    <!DOCTYPE html>
<html lang="ko">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>할 일 관리</title>
    <style>
        /* 전체 배경 */
        body {
            font-family: 'Segoe UI', sans-serif;
            background-color: #f0f0f0;
            display: flex;
            justify-content: center;
            padding-top: 50px;
            margin: 0;
        }

        /* 카드 영역 */
        .card {
            background: white;
            width: 100%;
            max-width: 500px;
            border-radius: 16px;
            box-shadow: 0 4px 20px rgba(0,0,0,0.1);
            padding: 30px;
        }

        /* 제목 */
        h1 {
            text-align: center;
            margin-top: 0;
            font-size: 1.6rem;
        }

        /* 입력 영역 */
        .input-area {
            display: flex;
            gap: 8px;
            margin-bottom: 20px;
        }

        .input-area input {
            flex: 7;

            border-radius: 8px;
            font-size: 1rem;
            outline: none;
        }

        .input-area input:focus {
            border-color: #4a90d9;
        }

        .input-area button {

        }

        /* 목록 */
        ul {

        ul li {

            align-items: center;
        }
    </style>
</head>
<body>
    <div class="card">
        <h1>📋 할 일 관리</h1>

        <div class="input-area">
            <input type="text" id="todo-input" placeholder="할 일을 입력하세요">
            <button id="add-btn">추가</button>
        </div>

        <ul id="todo-list">
            <!-- 할 일 항목이 여기에 추가됩니다 -->
        </ul>
    </div>

    <script>
        // HTML 요소 가져오기
        const input = document.getElementById('todo-input');
        const addBtn = document.getElementById('add-btn');
        const todoList = document.getElementById('todo-list');

        // 할 일을 추가하는 함수
        function addTodo() {
            const text = input.value.trim();

            // 빈 칸 확인
            if (text === '') {
                alert('할 일을 입력해주세요!');
                return;
            }

            // 새로운 항목(li) 만들기
            const li = document.createElement('li');
            li.textContent = text;

            // 목록에 추가
            todoList.appendChild(li);

            // 입력창 비우고 포커스
            input.value = '';
            input.focus();
        }

        // 버튼 클릭 시 추가
        addBtn.addEventListener('click', addTodo);

        // Enter 키 입력 시 추가
        input.addEventListener('keypress', function(e) {
            if (e.key === 'Enter') {
                addTodo();
            }
        });
    </script>
</body>
</html>

    ⚠️ 자주 하는 실수¶

    실수 1: AI에게 모든 기능을 한 번에 요청하기 🚫

    "로그인, 게시판, 댓글, 좋아요, 검색 기능이 있는 웹앱을 만들어줘"처럼 한 번에 요청하면 코드가 방대해지고, 오류 발견이 어렵습니다. 한 번에 하나의 블록만 요청하세요.

    실수 2: 에러 메시지를 무시하고 "안 돼요"만 전달하기 🚫

    콘솔의 빨간 에러 메시지는 오류의 위치와 원인을 알려주는 보물입니다. 반드시 복사하여 AI에게 전달하세요. 에러 메시지 없이는 AI도 추측만 할 수 있습니다.

    실수 3: 수정된 코드의 일부만 교체하기 🚫

    AI가 수정 코드를 줄 때, 어디를 바꿔야 할지 헷갈리면 전체 코드를 통째로 교체하는 것이 안전합니다. 일부만 바꾸다가 괄호가 빠지거나 코드 순서가 꼬이는 경우가 많습니다.

    실수 4: 이전 코드를 보내지 않고 추가 기능만 요청하기 🚫

    "이제 삭제 기능을 추가해줘"만 말하면, AI는 이전 코드의 구조를 모릅니다. 항상 현재까지의 전체 코드를 함께 보내세요.

    ✅ 스스로 점검하기¶

    기본 문제: 아래 구현 지시문의 문제점을 찾아보세요.

    게시판을 만들어줘. 글 쓰기, 수정, 삭제, 댓글, 좋아요, 
검색, 카테고리 분류, 페이지네이션 기능 포함.

    💡 답: 한 번에 너무 많은 기능을 요청하고 있습니다. 기술 스택, 화면 구성, 각 기능의 구체적 동작 설명이 없습니다. "블록 1: 글 목록을 보여주는 화면"부터 시작해야 합니다.

    응용 문제: 아래 에러 메시지를 보고, AI에게 보낼 디버깅 요청을 작성해보세요.

    Uncaught ReferenceError: addToDo is not defined
    at HTMLButtonElement.onclick (index.html:28)

    💡 답 예시: "버튼을 클릭하면 addToDo is not defined 에러가 발생합니다. 28번째 줄의 onclick에서 addToDo 함수를 호출하는데, 함수 이름이 일치하지 않는 것 같습니다. 코드에서 함수 이름은 addTodo입니다(대소문자 차이). 함수 이름을 통일해주세요."

    심화 문제: 여러분 팀의 웹앱에서 가장 핵심이 되는 기능 1가지를 골라, 블록으로 나누고 첫 번째 블록의 구현 지시문을 작성해보세요. 아래 항목을 포함해야 합니다.

    • 기술 스택 (HTML, CSS, JavaScript)
    • 화면에 보이는 요소들의 배치
    • 사용자가 하는 행동과 그 결과
