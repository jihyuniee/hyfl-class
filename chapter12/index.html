    Chapter 12. 불편함을 문제로 바꾸는 힘 – 문제 정의와 사용자 분석¶

    Part 4. 문제 발견과 알고리즘 설계 – 프로젝트의 뼈대 세우기

    🎯 이 장에서 배우는 것¶

    • 막연한 불편함을 '누가, 언제, 왜 불편한지'로 구체화하는 문제 정의 문장을 작성할 수 있다
    • 사용자 인터뷰와 설문을 설계하여 실제 니즈를 파악할 수 있다
    • 수집한 데이터를 AI와 함께 분석하여 핵심 문제를 도출할 수 있다
    • 팀원과 협의하여 프로젝트 주제를 하나로 선정할 수 있다

    💡 왜 이걸 배우나요?¶

    여러분이 감기에 걸려 병원에 갔다고 상상해보세요. 의사에게 "아파요"라고만 말하면 어떻게 될까요? 의사는 진단을 내릴 수 없습니다. 어디가, 언제부터, 어떻게 아픈지 물어본 후에야 비로소 처방을 내리죠.

    소프트웨어 프로젝트도 마찬가지입니다. "학교생활이 불편해요"라는 막연한 느낌으로는 아무것도 만들 수 없습니다. 하지만 "등교 전 날씨를 확인하지 못해 우산을 안 가져오는 학생이 주 2회 이상 비를 맞는다"라고 정의하면? 그제야 해결책이 보이기 시작합니다.

    이 장에서는 불편함이라는 증상을 정확한 진단으로 바꾸는 힘을 기릅니다. 이것이 모든 성공적인 프로젝트의 출발점입니다.

    📚 핵심 개념¶

    개념 1: 문제 정의 — 불편함에 이름 붙이기¶

    🔵 비유로 시작

    탐정이 사건을 수사할 때를 떠올려보세요. "뭔가 이상하다"는 직감만으로는 사건을 해결할 수 없습니다. 누가 피해자인지, 언제 사건이 발생했는지, 어디서 일어났는지, 왜 그런 일이 생겼는지를 하나씩 밝혀야 합니다. 문제 정의란 바로 이 탐정의 수사 보고서와 같습니다.

    🔵 정확한 정의

    문제 정의(Problem Definition): 해결하려는 문제의 대상(누가), 상황(언제/어디서), 핵심 원인(왜), 그리고 그로 인한 영향(어떤 결과)을 명확하게 서술한 문장

    🔵 예시로 확인

    나쁜 문제 정의와 좋은 문제 정의를 비교해봅시다.

    graph LR A["❌ 나쁜 문제 정의"] --> B["급식이 불편하다"] C["✅ 좋은 문제 정의"] --> D["우리 학교 1학년 학생들이<br/>점심시간에 급식 메뉴를<br/>미리 확인할 수 없어서<br/>알레르기 유발 음식을<br/>모르고 먹게 되는 경우가 있다"]

    좋은 문제 정의에는 다섯 가지 요소가 들어갑니다.

    • 누가(Who): 우리 학교 1학년 학생들
    • 언제(When): 점심시간에
    • 무엇이(What): 급식 메뉴를 미리 확인할 수 없어서
    • 왜 문제인지(Why): 알레르기 유발 음식을 모르고 먹게 됨
    • 얼마나(How much): 월 평균 2~3회 발생

    🔍 알아보기: 이 다섯 가지 요소를 기억하는 쉬운 방법이 있습니다. 뉴스 기사의 5W1H(누가, 언제, 어디서, 무엇을, 왜, 어떻게)와 비슷하죠? 기자가 기사를 쓸 때도 이 요소가 빠지면 독자가 상황을 이해할 수 없는 것처럼, 문제 정의도 마찬가지입니다.

    개념 2: 사용자 분석 — 진짜 사용자의 목소리 듣기¶

    🔵 비유로 시작

    신발 가게 주인이 새 신발을 디자인한다고 합시다. 자기 발에만 맞춰 만들면 어떻게 될까요? 다른 고객은 아무도 사지 않을 겁니다. 신발을 실제로 신을 사람들에게 "발이 어디가 불편한지", "어떤 상황에서 신을 건지" 물어봐야 합니다. 이것이 바로 사용자 분석입니다.

    🔵 정확한 정의

    사용자 분석(User Analysis): 프로그램을 실제로 사용할 사람들이 누구인지 파악하고, 그들의 특성·필요·행동 패턴을 조사하는 과정

    🔵 예시로 확인

    "급식 알레르기 확인 앱"을 만든다면, 사용자는 누구일까요?

    graph TD A["🎯 주요 사용자"] --> B["학생<br/>매일 급식을 먹는 당사자"] A --> C["학부모<br/>자녀의 알레르기를 걱정"] A --> D["영양사<br/>급식 메뉴를 관리"] B --> E["니즈: 오늘 메뉴에<br/>내 알레르기 식품이<br/>있는지 빠르게 확인"] C --> F["니즈: 자녀가 먹을<br/>메뉴를 미리 파악"] D --> G["니즈: 알레르기 관련<br/>문의를 줄이고 싶음"]

    모든 사용자가 똑같이 중요하지는 않습니다. 주요 사용자(Primary User)를 정하고, 그 사람의 니즈에 집중하는 것이 핵심입니다.

    개념 3: 설문과 인터뷰 — 데이터로 확인하기¶

    🔵 비유로 시작

    의사가 환자를 진단할 때, 환자의 말만 듣지 않습니다. 체온계로 열을 재고, 혈압을 측정하고, 엑스레이를 찍죠. "아픈 것 같아요"라는 주관적 느낌을 객관적 데이터로 확인하는 겁니다. 설문과 인터뷰는 우리의 "체온계"입니다.

    🔵 정확한 정의

    사용자 설문(Survey): 많은 사람에게 동일한 질문을 던져 패턴을 파악하는 정량적 조사 방법

    사용자 인터뷰(Interview): 소수의 사용자와 깊이 있는 대화를 통해 숨겨진 니즈를 발견하는 정성적 조사 방법

    🔵 좋은 질문 vs 나쁜 질문

    graph LR subgraph "❌ 나쁜 질문" A1["급식 앱 있으면 좋겠죠?"] A2["급식이 불편하지 않나요?"] end subgraph "✅ 좋은 질문" B1["오늘 급식 메뉴를<br/>어떻게 확인하나요?"] B2["급식 때문에 불편했던<br/>구체적 경험이 있나요?"] end

    나쁜 질문은 답을 유도합니다("네"라고 답할 수밖에 없죠). 좋은 질문은 사용자 스스로 경험을 이야기하게 만듭니다.

    💡 생각해보기: "이 앱을 쓰시겠습니까?"라는 질문에 "네"라고 답한 사람 중 실제로 앱을 다운로드하는 비율은 얼마나 될까요? 연구에 따르면 10~20%에 불과합니다. 그래서 미래의 의도가 아니라 과거의 행동을 묻는 것이 중요합니다.

    개념 4: AI와 함께하는 데이터 분석과 주제 선정¶

    🔵 비유로 시작

    퍼즐 조각 100개를 탁자 위에 펼쳐놓았다고 상상해보세요. 혼자서 하나씩 맞추면 오래 걸리지만, 옆에서 누군가 "이 조각은 하늘 부분이야", "이 색깔끼리 모아봐"라고 도와주면 훨씬 빠릅니다. AI는 설문 데이터를 분류하고 패턴을 찾아주는 퍼즐 도우미입니다.

    🔵 정확한 정의

    데이터 기반 주제 선정: 직감이나 선호가 아니라, 수집한 사용자 데이터의 패턴을 근거로 프로젝트 주제를 결정하는 방법

    🔵 AI 활용 프롬프트 예시

    AI에게 설문 결과를 분석해달라고 요청할 때, 이렇게 구체적으로 물어보세요.

    다음은 우리 반 30명에게 "학교생활에서 가장 불편한 점"을 물어본 설문 결과입니다.

[설문 결과 데이터 붙여넣기]

1. 가장 많이 언급된 불편함 상위 3개를 정리해줘
2. 각 불편함의 공통 원인이 무엇인지 분석해줘
3. 이 중 중학생이 프로그래밍으로 해결 가능한 문제는 무엇인지 추천해줘

    🔨 따라하기: 문제 정의서 완성 프로젝트¶

    이제 실제로 불편함을 문제 정의로 바꿔봅시다. 이전 장에서 브레인스토밍했던 불편함 목록을 꺼내세요!

    Step 1: 불편함 목록 다시 꺼내기¶

    이전에 작성한 불편함 목록이 없다면, 지금 빠르게 5개를 적어보세요.

    # 불편함 목록을 코드로 정리해봅시다
discomforts = [
    "아침에 오늘 시간표를 확인하기 어렵다",
    "급식 메뉴에 알레르기 식품이 있는지 모른다",
    "도서관 책 반납일을 자꾸 잊어버린다",
    "모둠 과제할 때 역할 분담이 불공평하다",
    "체육복을 빌려야 할 때 누구에게 빌릴지 모른다"
]

print("=== 나의 불편함 목록 ===")
for i, item in enumerate(discomforts, 1):
    print(f"{i}. {item}")

    === 나의 불편함 목록 ===
1. 아침에 오늘 시간표를 확인하기 어렵다
2. 급식 메뉴에 알레르기 식품이 있는지 모른다
3. 도서관 책 반납일을 자꾸 잊어버린다
4. 모둠 과제할 때 역할 분담이 불공평하다
5. 체육복을 빌려야 할 때 누구에게 빌릴지 모른다

    Step 2: 불편함에 5W 질문 던지기¶

    각 불편함을 구체적인 문제로 바꾸기 위해, 파이썬 딕셔너리로 5W를 정리합니다.

    # 하나의 불편함을 골라 5W로 분석합니다
problem_analysis = {
    "불편함": "도서관 책 반납일을 자꾸 잊어버린다",
    "누가(Who)": "우리 학교 학생 (특히 책을 자주 빌리는 학생)",
    "언제(When)": "반납일 당일 아침 또는 반납일이 지난 후",
    "어디서(Where)": "학교 도서관 대출/반납 과정에서",
    "무엇이(What)": "반납일을 기억하지 못해 연체가 발생",
    "왜(Why)": "별도의 알림 시스템이 없고, 대출증에 적힌 날짜를 확인하지 않음"
}

print("=== 5W 문제 분석 ===")
for key, value in problem_analysis.items():
    print(f"  {key}: {value}")

    === 5W 문제 분석 ===
  불편함: 도서관 책 반납일을 자꾸 잊어버린다
  누가(Who): 우리 학교 학생 (특히 책을 자주 빌리는 학생)
  언제(When): 반납일 당일 아침 또는 반납일이 지난 후
  어디서(Where): 학교 도서관 대출/반납 과정에서
  무엇이(What): 반납일을 기억하지 못해 연체가 발생
  왜(Why): 별도의 알림 시스템이 없고, 대출증에 적힌 날짜를 확인하지 않음
