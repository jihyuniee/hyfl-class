    Chapter 15. 알고리즘 설계 실전 — 순서도와 의사결정 트리¶

    Part 5 | [2권] 컴퓨팅 사고력 — 문제를 분해하고 알고리즘으로 설계하기

    🎯 이 장에서 배우는 것¶

    • 순서도의 기본 기호를 이해하고, 실생활 문제를 순서도로 시각화할 수 있다
    • 조건 분기(만약 ~라면)와 반복 구조(~하는 동안)를 활용한 알고리즘을 설계할 수 있다
    • 의사결정 트리의 구조를 이해하고, 분류·추천 문제에 적용할 수 있다
    • 설계한 알고리즘의 허점을 발견하고 개선할 수 있다

    💡 왜 이걸 배우나요?¶

    여러분은 오늘 아침에도 수많은 결정을 내렸습니다.

    "알람이 울렸다 → 일어날까, 5분만 더 잘까? → 비가 오나? → 우산을 챙길까? → 버스가 빠를까, 걸어갈까?"

    이 모든 과정이 사실은 알고리즘입니다. 머릿속에 있는 이 알고리즘을 눈에 보이는 그림으로 바꿀 수 있다면 어떨까요?

    순서도(플로우차트)와 의사결정 트리는 "생각을 그림으로 옮기는 도구"입니다. 프로그래머가 코드를 짜기 전에 설계도를 그리듯, 복잡한 문제도 순서도로 정리하면 빠짐없이, 논리적으로 해결할 수 있습니다. 앱 개발자, 데이터 과학자, 의사, 경영자 등 다양한 분야에서 매일 사용하는 핵심 사고 도구입니다.

    이 장을 마치면, 여러분은 어떤 문제든 "그림으로 정리하고 → 코드로 옮기는" 능력을 갖추게 됩니다.

    📚 핵심 개념¶

    개념 1: 순서도(플로우차트)의 기본 기호¶

    1. 비유로 시작¶

    순서도는 "보물 지도"와 같습니다. 보물 지도에 출발점, 갈림길, 목적지가 표시되어 있듯이 순서도에는 시작, 판단, 처리, 끝이 기호로 표시됩니다. 지도를 따라가면 누구나 보물을 찾을 수 있듯, 순서도를 따라가면 누구나 같은 결과에 도달합니다.

    2. 정확한 정의¶

    순서도(Flowchart) 란 알고리즘의 각 단계를 약속된 도형 기호와 화살표로 나타낸 시각적 다이어그램입니다. 국제 표준(ISO 5807)으로 기호가 정해져 있어, 전 세계 누구나 같은 방식으로 읽을 수 있습니다.

    핵심 기호 4가지:

    • 타원(둥근 사각형) — 시작/끝: 알고리즘의 출발과 종료 지점
    • 직사각형 — 처리(Process): 계산하거나 행동하는 단계 (예: "우산을 챙긴다")
    • 마름모 — 판단(Decision): 예/아니오로 나뉘는 질문 (예: "비가 오는가?")
    • 화살표 — 흐름: 단계와 단계를 연결하는 방향 표시

    3. 예시로 확인 — "아침 등교 알고리즘"¶

    flowchart TD A([🔔 알람이 울린다]) --> B[일어나서 씻는다] B --> C{비가 오는가?} C -->|예| D[우산을 챙긴다] C -->|아니오| E[우산 없이 준비한다] D --> F[집을 나선다] E --> F F --> G{버스가 5분 내 오는가?} G -->|예| H[버스를 탄다] G -->|아니오| I[걸어서 간다] H --> J([🏫 학교 도착]) I --> J

    📌 알아보기: 마름모 기호에서 나오는 화살표는 반드시 두 갈래 이상이어야 합니다. "예"와 "아니오"처럼 조건의 결과를 명확히 표시해야 다른 사람이 읽었을 때 혼동이 없습니다.

    개념 2: 조건 분기 — "만약 ~라면" 구조¶

    1. 비유로 시작¶

    조건 분기는 "갈림길의 이정표"입니다. 등산로에서 "체력이 충분하면 정상 코스, 아니면 둘레길"이라는 안내판을 만나는 것과 같습니다. 조건에 따라 서로 다른 길로 진행합니다.

    2. 정확한 정의¶

    조건 분기(Conditional Branch) 란 주어진 조건이 참(True)인지 거짓(False)인지에 따라 실행 경로가 달라지는 알고리즘 구조입니다.

    두 가지 형태:

    • 단일 조건 분기 (if-else): 조건 하나로 두 갈래
    • 다중 조건 분기 (if-elif-else): 여러 조건을 순서대로 검사

    3. 예시로 확인 — "시험 점수에 따른 등급 판정"¶

    flowchart TD A([시작]) --> B[점수를 입력받는다] B --> C{점수 ≥ 90?} C -->|예| D[등급 = A] C -->|아니오| E{점수 ≥ 80?} E -->|예| F[등급 = B] E -->|아니오| G{점수 ≥ 70?} G -->|예| H[등급 = C] G -->|아니오| I[등급 = D] D --> J[등급을 출력한다] F --> J H --> J I --> J J --> K([끝])

    💬 생각해보기: 위 순서도에서 점수가 85점이라면, 화살표를 따라 어떤 경로로 이동하는지 손가락으로 짚어보세요. C에서 "아니오" → E에서 "예" → 등급 B가 됩니다!

    개념 3: 반복 구조 — "~하는 동안" 되풀이하기¶

    1. 비유로 시작¶

    반복 구조는 "빨래 바구니 비우기"와 같습니다. 바구니에 빨래가 남아있는 동안 한 벌씩 꺼내서 널고, 바구니가 비면 멈춥니다. 몇 벌인지 모르지만, "비었나?" 확인하면서 반복합니다.

    2. 정확한 정의¶

    반복 구조(Loop/Iteration) 란 특정 조건이 만족되는 동안 같은 작업을 되풀이하는 알고리즘 구조입니다.

    두 가지 핵심 패턴:

    • 조건 반복 (while): "조건이 참인 동안" 반복 → 몇 번 반복할지 모를 때
    • 횟수 반복 (for): "정해진 횟수만큼" 반복 → 몇 번인지 알 때

    3. 예시로 확인 — "교실 30명 출석 체크"¶

    flowchart TD A([시작]) --> B["학생번호 = 1, 출석인원 = 0"] B --> C{학생번호 ≤ 30?} C -->|아니오| H["출석인원을 출력한다"] C -->|예| D["해당 학생의 이름을 부른다"] D --> E{대답했는가?} E -->|예| F["출석인원 = 출석인원 + 1"] E -->|아니오| G["결석으로 기록한다"] F --> I["학생번호 = 학생번호 + 1"] G --> I I --> C H --> J([끝])

    📌 알아보기: 반복 구조에서 가장 중요한 것은 "반복을 멈추는 조건"입니다. 위 예시에서는 학생번호가 30을 넘으면 멈춥니다. 이 멈추는 조건이 없으면 무한 반복(무한 루프)에 빠져 알고리즘이 영원히 끝나지 않습니다!

    개념 4: 의사결정 트리 — 질문을 따라가면 답이 나온다¶

    1. 비유로 시작¶

    의사결정 트리는 "스무고개 게임"과 같습니다. "동물인가요?" → "다리가 4개인가요?" → "집에서 키울 수 있나요?" 질문을 하나씩 따라가면 최종 답에 도달합니다. 나무(Tree)가 뿌리에서 가지를 뻗어나가듯, 질문이 가지를 이루며 뻗어갑니다.

    2. 정확한 정의¶

    의사결정 트리(Decision Tree) 란 일련의 질문(조건)을 나무 구조로 배열하여, 질문에 대한 답을 따라가면 최종 결론(분류 또는 결정)에 도달하는 알고리즘입니다.

    구성 요소:

    • 루트 노드(Root): 가장 첫 번째 질문 (나무의 뿌리)
    • 내부 노드(Internal Node): 중간 질문들 (가지가 갈라지는 지점)
    • 리프 노드(Leaf): 최종 결론/답 (나무의 잎)
    • 가지(Branch): 질문의 답에 따른 연결선

    3. 예시로 확인 — "오늘 뭐 먹을까? 메뉴 추천"¶

    flowchart TD A{"🍽️ 따뜻한 음식이<br/>먹고 싶나?"} A -->|예| B{"매운 것을<br/>좋아하나?"} A -->|아니오| C{"디저트류를<br/>원하나?"} B -->|예| D["🍜 짬뽕 추천!"] B -->|아니오| E["🍲 된장찌개 추천!"] C -->|예| F["🍦 아이스크림 추천!"] C -->|아니오| G["🥗 샐러드 추천!"]

    순서도와 의사결정 트리는 비슷해 보이지만 목적이 다릅니다.

    순서도의 특징: - 작업의 순서와 흐름을 보여주는 데 초점 - 반복, 처리, 입출력 등 다양한 동작 포함 - "이 일을 어떻게 처리하지?"라는 질문에 답함

    의사결정 트리의 특징: - 분류와 선택에 초점 - 질문-답변의 연쇄로 결론 도달 - "이것은 어떤 종류이지?" 또는 "무엇을 선택하지?"라는 질문에 답함

    💬 생각해보기: 병원에서 의사가 환자를 진단하는 과정을 떠올려 보세요. "열이 있나요?" → "기침을 하나요?" → "며칠 됐나요?" 이런 질문의 연쇄가 바로 의사결정 트리입니다. 실제로 의료 분야에서 진단 트리가 활발히 사용됩니다!

    🔨 따라하기: "분실물 찾기 알고리즘" 설계하기¶

    학교에서 물건을 잃어버렸을 때, 체계적으로 찾는 알고리즘을 순서도로 설계하고 Python 코드로 구현해 봅시다.

    Step 1: 문제 분석 — 무엇을 해결할 것인가?¶

    먼저 "분실물을 찾는 과정"을 일상 언어로 정리합니다.

    문제 정의: 학생이 물건을 잃어버렸을 때, 어디서 찾아야 할지 단계적으로 안내하는 시스템을 만든다.

    입력: 잃어버린 물건의 종류, 마지막으로 본 장소

    출력: 찾아볼 장소 안내 → 찾았는지 확인 → 분실물 센터 안내

    먼저 자연어(일상 말)로 단계를 나열합니다:

    1. 잃어버린 물건이 무엇인지 확인한다
    2. 마지막으로 본 장소를 떠올린다
    3. 그 장소를 확인한다
    4. 찾았으면 끝, 못 찾았으면 다음 단계로
    5. 자주 가는 장소 목록을 하나씩 확인한다
    6. 그래도 못 찾았으면 분실물 센터에 신고한다

    Step 2: 순서도 설계 — 생각을 그림으로¶

    위의 자연어 단계를 순서도 기호로 변환합니다.

    flowchart TD A([시작]) --> B["물건 종류를 입력받는다"] B --> C["마지막으로 본 장소를 입력받는다"] C --> D["해당 장소를 확인한다"] D --> E{물건을 찾았는가?} E -->|예| F["🎉 축하합니다! 물건을 찾았습니다"] F --> L([끝]) E -->|아니오| G["장소 목록 = 교실, 급식실,<br/>도서관, 운동장, 화장실"] G --> H{"확인할 장소가<br/>남아있는가?"} H -->|예| I["다음 장소를 확인한다"] I --> J{물건을 찾았는가?} J -->|예| F J -->|아니오| H H -->|아니오| K["📋 분실물 센터에 신고합니다"] K --> L

    📌 알아보기: 이 순서도에는 조건 분기(마름모 "찾았는가?")와 반복(장소 목록을 하나씩 확인)이 모두 들어있습니다. 두 가지 핵심 구조를 한 번에 연습할 수 있는 좋은 예제입니다.

    Step 3: 의사코드(Pseudocode)로 변환¶

    순서도를 코드로 바로 옮기기 전에, 의사코드(프로그래밍 언어와 비슷하지만 사람이 읽기 쉬운 형태)로 중간 단계를 거칩니다.

    시작
    물건 = 입력("잃어버린 물건은?")
    마지막장소 = 입력("마지막으로 본 장소는?")

    출력(마지막장소 + "을(를) 확인하세요")
    결과 = 입력("찾았나요? (예/아니오)")

    만약 결과 == "예":
        출력("축하합니다! 물건을 찾았습니다!")
    아니면:
        장소목록 = [교실, 급식실, 도서관, 운동장, 화장실]
        찾음 = 거짓

    💬 생각해보기: 의사코드는 왜 필요할까요? 순서도는 전체 흐름을 한눈에 보여주고, 의사코드는 코드와 가까운 형태로 논리를 정리해 줍니다. 순서도 → 의사코드 → 실제 코드, 이 3단계 습관을 들이면 복잡한 프로그램도 실수 없이 만들 수 있습니다.

    Step 4: Python 코드로 구현¶

    이제 의사코드를 실제 Python 코드로 변환합니다. 한 줄 한 줄 의사코드와 대응시키며 읽어보세요.

    # === 분실물 찾기 알고리즘 === #

def find_lost_item():
    """분실물을 단계적으로 찾아주는 프로그램"""

    # 1단계: 정보 수집
    item = input("🔍 잃어버린 물건은 무엇인가요? ")
    last_place = input("📍 마지막으로 본 장소는 어디인가요? ")

    # 2단계: 마지막 장소 확인
    print(f"\n👉 먼저 [{last_place}]을(를) 확인해 보세요!")
    result = input("찾았나요? (예/아니오): ")

    if result == "예":
        print(f"\n🎉 축하합니다! [{item}]을(를) 찾았네요!")
        return  # 함수 종료

    # 3단계: 주요 장소 순회 (반복 구조)
    places = ["교실", "급식실", "도서관", "운동장", "화장실"]
    found = False  # 찾았는지 여부를 추적하는 변수

    print(f"\n😥 [{last_place}]에 없군요. 다른 장소를 확인해 볼게요.")

    for place in places:
        print(f"\n👉 [{place}]을(를) 확인해 보세요!")
        result = input("찾았나요? (예/아니오): ")

        if result == "예":
            print(f"\n🎉 축하합니다! [{item}]을(를) [{place}]에서 찾았네요!")
            found = True
            break  # 반복 중단

    # 4단계: 최종 안내 (조건 분기)
    if not found:
        print(f"\n😢 모든 장소를 확인했지만 [{item}]을(를) 찾지 못했습니다.")
        print("📋 분실물 센터(학생부)에 신고해 주세요!")
        print(f"   - 분실물: {item}")
        print(f"   - 마지막 목격 장소: {last_place}")

# 프로그램 실행
find_lost_item()

    Step 5: 실행하고 테스트하기¶

    프로그램을 실행한 뒤, 다양한 시나리오로 테스트해 봅니다.

    테스트 시나리오 1 — 첫 번째 장소에서 바로 찾는 경우: - 물건: 체육복 / 마지막 장소: 체육관 / 찾았나요?: 예 - 예상 결과: "축하합니다!" 메시지 출력 후 종료

    테스트 시나리오 2 — 반복 도중 찾는 경우: - 물건: 필통 / 마지막 장소: 복도 / 복도에서: 아니오 / 교실: 아니오 / 급식실: 예 - 예상 결과: 급식실에서 찾았다는 메시지 출력

    테스트 시나리오 3 — 끝까지 못 찾는 경우: - 모든 장소에서 "아니오" 선택 - 예상 결과: 분실물 센터 신고 안내 출력

    📌 알아보기: 이렇게 여러 경우를 미리 정해놓고 테스트하는 것을 "테스트 케이스 작성"이라고 합니다. 프로 개발자들도 반드시 하는 과정입니다. 순서도의 모든 경로(화살표)를 최소 한 번씩은 지나가도록 테스트해야 빈틈이 없습니다.

    Step 6: 의사결정 트리 도전 — "동아리 추천 알고리즘"¶

    이번에는 의사결정 트리를 활용하여, 학생의 성향에 맞는 동아리를 추천하는 프로그램을 만들어 봅시다.

    먼저 의사결정 트리를 설계합니다:

    flowchart TD Q1{"🏃 몸을 움직이는<br/>활동을 좋아하나요?"} Q1 -->|예| Q2{"🤝 팀으로 하는 것을<br/>선호하나요?"} Q1 -->|아니오| Q3{"🎨 창작 활동에<br/>관심이 있나요?"} Q2 -->|예| R1["⚽ 축구부 추천!"] Q2 -->|아니오| R2["🏃 육상부 추천!"] Q3 -->|예| Q4{"✍️ 글쓰기와 그리기<br/>중 뭘 더 좋아하나요?"} Q3 -->|아니오| R3["💻 코딩부 추천!"] Q4 -->|글쓰기| R4["📝 문예창작부 추천!"] Q4 -->|그리기| R5["🎨 미술부 추천!"]

    이 트리를 Python 코드로 구현합니다:

    # === 동아리 추천 의사결정 트리 === #

def recommend_club():
    """의사결정 트리로 동아리를 추천하는 프로그램"""

    print("=" * 40)
    print("🎯 나에게 맞는 동아리 찾기!")
    print("=" * 40)

    # 첫 번째 질문 (루트 노드)
    q1 = input("\n🏃 몸을 움직이는 활동을 좋아하나요? (예/아니오): ")

    if q1 == "예":
        # 두 번째 질문 (내부 노드)
        q2 = input("🤝 팀으로 하는 것을 선호하나요? (예/아니오): ")

        if q2 == "예":
            print("\n⚽ 축구부를 추천합니다!")  # 리프 노드
        else:
            print("\n🏃 육상부를 추천합니다!")  # 리프 노드
    else:
        # 세 번째 질문 (내부 노드)
        q3 = input("🎨 창작 활동에 관심이 있나요? (예/아니오): ")

        if q3 == "예":
            q4 = input("✍️ 글쓰기와 그리기 중 뭘 더 좋아하나요? (글쓰기/그리기): ")

            if q4 == "글쓰기":
                print("\n📝 문예창작부를 추천합니다!")
            else:
                print("\n🎨 미술부를 추천합니다!")
        else:
            print("\n💻 코딩부를 추천합니다!")

    print("\n동아리 체험 주간에 직접 방문해 보세요! 😊")

# 프로그램 실행
recommend_club()
