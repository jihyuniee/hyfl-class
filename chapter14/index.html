    14. 화면을 설계하자 – 와이어프레임과 UI 기획¶

    🎯 이 장에서 배우는 것¶

    • 와이어프레임의 개념과 목적을 설명할 수 있다
    • UI(사용자 인터페이스)의 기본 요소를 구분할 수 있다
    • 사용자 흐름(User Flow)을 고려한 화면 구성을 기획할 수 있다
    • 구글 슬라이드(또는 HTML)를 활용해 자신의 웹앱 와이어프레임을 제작할 수 있다

    💡 왜 이걸 배우나요?¶

    여러분이 매일 사용하는 앱을 떠올려보세요. 카카오톡을 열면 채팅 목록이 바로 보이고, 유튜브를 열면 추천 영상이 한눈에 들어옵니다. 이런 화면 배치가 우연히 만들어졌을까요?

    절대 아닙니다.

    모든 앱과 웹사이트는 코드를 작성하기 전에 화면을 먼저 설계합니다. 마치 건물을 지을 때 벽돌을 쌓기 전에 설계도를 그리는 것처럼요.

    이 설계도를 와이어프레임(Wireframe)이라고 합니다.

    와이어프레임 없이 바로 코딩을 시작하면 어떻게 될까요?

    • "이 버튼 어디에 넣지?" → 고민하느라 시간 낭비 ⏰
    • "아, 이 화면도 필요했네!" → 뒤늦게 발견하고 처음부터 다시 작업 😫
    • "팀원이 상상한 화면이 나랑 다르네..." → 소통 실패 🤯

    이번 장에서 와이어프레임을 배우면, 여러분의 프로젝트가 체계적이고 효율적으로 진행될 수 있습니다.

    📚 핵심 개념¶

    개념 1: 와이어프레임 – 앱의 설계도 📐¶

    🏠 비유로 시작하기

    집을 짓는다고 상상해보세요.

    1. 먼저 설계도를 그립니다 → 방이 몇 개인지, 화장실 위치, 현관 방향
    2. 그다음 골조 공사를 합니다 → 뼈대를 세움
    3. 마지막에 인테리어를 합니다 → 벽지, 가구, 조명

    와이어프레임은 바로 1단계 설계도에 해당합니다. 예쁜 색상이나 폰트(인테리어)는 나중 일이고, 먼저 "어떤 요소가 어디에 배치되는가"를 잡는 것입니다.

    ✏️ 정확한 정의

    와이어프레임(Wireframe): 앱이나 웹사이트의 화면 구조를 단순한 선과 도형으로 표현한 설계도. 색상, 이미지 같은 시각 디자인 요소 없이 레이아웃과 기능 배치에 집중한다.

    📌 예시로 확인

    급식 앱의 와이어프레임을 떠올려보면:

    • 상단: 날짜 선택 영역
    • 중앙: 오늘의 메뉴 목록
    • 하단: 내일/이번 주 전환 버튼

    이것을 예쁜 디자인 없이, 회색 박스와 텍스트만으로 표현한 것이 와이어프레임입니다.

    graph LR A["🏠 건축의 설계도"] --- B["📱 앱의 와이어프레임"] A --- C["방 배치, 문 위치"] B --- D["버튼 위치, 화면 구성"]

    생각해보기: 여러분이 자주 쓰는 앱 하나를 골라보세요. 첫 화면에 어떤 요소들이 있나요? 그 요소들은 왜 그 위치에 있을까요?

    개념 2: UI의 기본 요소 – 화면을 구성하는 블록들 🧱¶

    🧩 비유로 시작하기

    레고를 조립할 때, 다양한 종류의 블록이 있죠? 긴 블록, 짧은 블록, 바퀴 블록, 창문 블록... 화면을 만드는 것도 마찬가지입니다. 몇 가지 기본 블록(UI 요소)을 조합해서 화면을 완성합니다.

    ✏️ 정확한 정의

    UI(User Interface, 사용자 인터페이스): 사용자가 앱이나 웹사이트와 상호작용하는 접점. 버튼을 누르고, 글자를 읽고, 정보를 입력하는 모든 화면 요소를 포함한다.

    📌 핵심 UI 요소 6가지

    • 헤더(Header): 화면 최상단. 앱 이름, 뒤로 가기 버튼 등이 위치
    • 내비게이션(Navigation): 다른 화면으로 이동하는 메뉴. 하단 탭 바가 대표적
    • 버튼(Button): 사용자가 누르면 동작이 실행되는 요소
    • 입력 필드(Input Field): 텍스트를 입력하는 칸 (검색창, 로그인 폼 등)
    • 카드(Card): 관련 정보를 묶어서 보여주는 박스 (게시물, 상품 정보 등)
    • 모달(Modal): 화면 위에 뜨는 팝업 창 (확인/취소 알림 등)
    graph TD UI["📱 화면 구성 요소"] UI --> H["🔝 헤더: 앱 이름, 뒤로가기"] UI --> N["📍 내비게이션: 탭 바, 메뉴"] UI --> B["👆 버튼: 동작 실행"] UI --> I["⌨️ 입력 필드: 텍스트 입력"] UI --> C["🃏 카드: 정보 묶음"] UI --> M["💬 모달: 팝업 알림"]

    알아보기: 카카오톡의 첫 화면에서 위 6가지 요소를 찾아보세요. 헤더에는 무엇이 있나요? 내비게이션은 어디에 있나요?

    개념 3: 사용자 흐름(User Flow) – 화면과 화면을 잇는 길 🗺️¶

    🛤️ 비유로 시작하기

    놀이공원에 갔다고 상상해보세요. 입구에서 출발해 → 매표소 → 놀이기구 선택 → 대기줄 → 탑승 → 출구로 나가는 동선이 있죠? 앱에도 이런 동선이 있습니다. 이것을 사용자 흐름(User Flow)이라고 합니다.

    ✏️ 정확한 정의

    사용자 흐름(User Flow): 사용자가 앱에서 특정 목표를 달성하기까지 거치는 화면과 행동의 순서. "어디서 시작해서, 어떤 화면을 거쳐, 어디서 끝나는가"를 나타낸다.

    📌 예시로 확인: 급식 확인 앱

    flowchart LR A["🏠 홈 화면"] -->|오늘 급식 터치| B["🍽️ 급식 상세"] A -->|날짜 변경 터치| C["📅 달력 화면"] C -->|날짜 선택| B B -->|뒤로가기| A A -->|설정 터치| D["⚙️ 설정 화면"]

    이 다이어그램을 보면, 사용자가 어디서 어디로 이동할 수 있는지 한눈에 파악됩니다.

    정리하기: 사용자 흐름을 먼저 그리면 ① 필요한 화면이 몇 개인지 알 수 있고 ② 빠뜨린 화면이 없는지 확인할 수 있고 ③ 사용자 입장에서 불편한 점을 미리 발견할 수 있습니다.

    개념 4: 좋은 UI vs 나쁜 UI – 사용자를 생각하는 설계 🎯¶

    🚪 비유로 시작하기

    밀어야 열리는 문에 손잡이가 달려있으면 어떨까요? 사람들은 본능적으로 손잡이를 당기려고 합니다. 이것이 나쁜 UI입니다. 반대로, 밀어야 하는 문에 밀판(넓은 판)이 달려있으면 자연스럽게 밀게 됩니다. 이것이 좋은 UI입니다.

    ✏️ 좋은 UI의 3가지 원칙

    • 명확성: 이 버튼을 누르면 무슨 일이 일어나는지 바로 알 수 있어야 한다
    • 일관성: 같은 종류의 버튼은 같은 위치, 같은 모양이어야 한다
    • 단순성: 한 화면에 너무 많은 정보를 넣지 않는다
    graph LR subgraph 좋은_UI["✅ 좋은 UI"] G1["명확한 버튼 텍스트"] G2["일관된 레이아웃"] G3["핵심 기능에 집중"] end subgraph 나쁜_UI["❌ 나쁜 UI"] B1["의미 모를 아이콘"] B2["매번 바뀌는 구조"] B3["한 화면에 모든 것"] end

    📌 실제 비교 예시

    ❌ 나쁜 UI의 급식 앱 - 첫 화면에 급식, 시간표, 공지, 날씨, 학사일정이 모두 빼곡히 들어있음 - 급식 보기 버튼이 "GO"라고만 적혀있음 - 뒤로 가기 버튼이 어떤 화면에서는 왼쪽 위, 어떤 화면에서는 오른쪽 아래

    ✅ 좋은 UI의 급식 앱 - 첫 화면에 "오늘의 급식"이 크게 보임 - 버튼에 "이번 주 급식 보기"라고 명확히 적혀있음 - 뒤로 가기 버튼이 항상 왼쪽 위에 위치

    🔨 따라하기: 나만의 와이어프레임 만들기¶

    이제 실제로 와이어프레임을 만들어봅시다! 여기서는 "우리 학교 급식 앱"을 예제로 사용합니다. 여러분은 자신의 프로젝트 주제로 대체해도 좋습니다.

    Step 1: 필요한 화면 목록 정하기 📋¶

    먼저, 앱에 어떤 화면들이 필요한지 목록을 만듭니다.

    생각해보기: "사용자가 이 앱으로 무엇을 하고 싶을까?"를 기준으로 생각하세요.

    급식 앱 예시:

    • 화면 1 – 홈 (오늘의 급식을 바로 보여줌)
    • 화면 2 – 주간 급식 (이번 주 급식 전체)
    • 화면 3 – 급식 상세 (영양 정보, 알레르기 정보)
    • 화면 4 – 설정 (알레르기 필터 설정)

    💡 팁: 처음부터 화면을 10개씩 만들려고 하지 마세요. 핵심 기능에 필요한 3~5개 화면이면 충분합니다.

    Step 2: 사용자 흐름 그리기 🗺️¶

    화면 목록이 정해졌으면, 화면 사이의 이동 경로를 그립니다.

    종이에 손으로 그려도 되고, 아래처럼 정리해도 됩니다.

    flowchart TD HOME["🏠 홈 화면<br/>오늘의 급식"] -->|이번 주 보기 터치| WEEK["📅 주간 급식"] HOME -->|메뉴 터치| DETAIL["🍽️ 급식 상세<br/>영양·알레르기 정보"] WEEK -->|날짜 터치| DETAIL HOME -->|설정 아이콘 터치| SETTING["⚙️ 설정<br/>알레르기 필터"] DETAIL -->|뒤로| HOME WEEK -->|뒤로| HOME SETTING -->|뒤로| HOME

    이렇게 그려놓으면 "아, 주간 급식에서 상세로 갈 수 있어야 하는구나!" 같은 것을 미리 발견할 수 있습니다.

    Step 3: 화면별 요소 배치하기 ✏️¶

    각 화면에 어떤 UI 요소가 들어가는지 글로 정리합니다.

    홈 화면 구성: - 🔝 헤더: 앱 이름 "우리학교 급식", 설정 아이콘 - 📅 날짜 표시: "2025년 1월 15일 (수)" - 🍽️ 급식 카드: 메뉴 항목 리스트 (밥, 국, 반찬 등) - 👆 버튼: "이번 주 급식 보기" - 📍 하단 내비게이션: 홈 / 주간 / 설정

    Step 4: HTML로 와이어프레임 제작하기 💻¶

    Step 5: 브라우저에서 확인하고 수정하기 🔍¶

    파일을 저장한 후 브라우저에서 열면, 3개의 스마트폰 모양 화면이 나란히 보입니다.

    확인할 사항:

    • 각 화면에 필요한 요소가 다 있는가?
    • 사용자가 다음 화면으로 어떻게 이동하는지 명확한가?
    • 한 화면에 정보가 너무 많지는 않은가?

    수정이 필요하면 HTML 코드를 직접 고쳐보세요. 예를 들어: - 메뉴 항목을 추가하고 싶으면 menu-item 클래스의 div를 복사 - 새 화면을 추가하고 싶으면 phone-frame 블록을 복사해서 내용 수정
